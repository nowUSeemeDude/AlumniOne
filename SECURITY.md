# Security Policy

## Supported Versions

This project actively maintains security updates for the following versions.

| Version                   | Supported |
| ------------------------- | --------- |
| Latest Release            | Yes       |
| Development (main branch) | Yes       |
| Older versions            | No        |

Security fixes are applied to the latest stable release and the active development branch.

---

## Reporting a Vulnerability

The security of this project is taken seriously. If you discover a vulnerability, please report it responsibly.

Do **not** open a public GitHub issue for security vulnerabilities.

Instead, report the issue privately by contacting the maintainers.

### How to Report

Send a report including the following information:

* Description of the vulnerability
* Steps to reproduce the issue
* Potential impact of the vulnerability
* Screenshots, logs, or proof-of-concept if available
* Suggested mitigation (if known)

Submit your report via:

Email: **[rah.ashikin@gmail.com](mailto:rah.ashikin@gmail.com)**

If email is not available, you may contact the repository maintainer directly through GitHub.

---

## Response Timeline

After a vulnerability report is submitted:

* **Acknowledgement:** within 48 hours
* **Initial assessment:** within 5 business days
* **Fix development:** depending on severity
* **Public disclosure:** after a patch is released

Critical vulnerabilities may be prioritized for immediate resolution.

---

## Responsible Disclosure

We ask that researchers and contributors follow responsible disclosure practices:

* Allow time for the issue to be investigated and resolved before public disclosure
* Do not exploit the vulnerability beyond what is necessary to demonstrate the issue
* Do not access or modify other users’ data

We appreciate responsible reporting and will acknowledge contributors who help improve the security of the project.

---

## Security Best Practices for Contributors

To maintain a secure codebase, contributors should:

* Avoid committing secrets or API keys
* Follow secure authentication practices
* Validate user input and sanitize outputs
* Keep dependencies up to date
* Follow secure coding standards

Pull requests may be reviewed with security considerations in mind.

---

## Security Updates

Security updates and patches will be announced through:

* GitHub releases
* Repository changelog
* Project documentation

Users are encouraged to keep their installations updated to the latest version.
